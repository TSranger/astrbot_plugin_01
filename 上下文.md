# AstrBot Plugin 01 上下文

更新时间：2026-06-27

## 插件定位

`astrbot_plugin_01` 是 AstrBot 的群聊增强插件，目标是在群聊里同时承担以下角色：

- 稳定响应 @、点名、直接问话、引用、followup
- 解析文本、图片、表情包、引用消息，并把图片内容纳入上下文
- 维护长期记忆、短窗口接话、主动插话和多层摘要记忆
- 定时执行主动发言任务，包括新闻自拍、常规主动说话和记忆归纳
- 使用 skill 作为角色与外观来源，当前默认 skill 由配置决定

## 目录结构

主要文件如下：

- `main.py`
  - 插件主入口
  - 群消息入口、记忆、回复、主动插话、图片识别、调度、日志
- `news_selfie.py`
  - 新闻抓取、新闻筛选、文章抽取、图片下载
  - 新闻自拍文案生成、图片生成、skill 外观读取、图片合成
- `llm_router.py`
  - 插件内部 LLM 路由层
  - 支持文本和视觉输入，必要时把本地图片转成 `data:` URL
- `config.yaml`
  - 插件全部配置
  - 包括群范围、skill、图片识别、回复发送、主动发言、新闻自拍等参数
- `db_manager.py`
  - 长期记忆与调度状态的 SQLite 管理

## 基本功能

### 1. 群消息处理

`main.py` 中的 `on_group_message()` 是核心入口，负责：

- 校验群白名单和插件总开关
- 从消息链提取正文、图片、表情、引用
- 通过视觉模型描述图片，再拼回消息上下文
- 构造触发状态，判断是否需要即时回复或记忆更新
- 处理 followup、短窗口接话、主动插话等逻辑

当前入口更强调“只处理真正的消息”，避免非消息事件混入正文。

### 2. 图片识别

图片识别链路位于 `main.py`，现状是：

- 支持多张图片并发识别
- 优先把本地可读图片转成 `data:` URL 交给视觉模型
- prompt 强调 OCR、图中文字、梗图语义和整体语境
- 默认图片描述长度已经调高，不再只产出极短概括

### 3. 记忆系统

插件内部维护了多层记忆与摘要：

- paragraph / daily / month / year / history 分层归纳
- 事件记忆与动态事件压缩
- 时间回忆和事件回忆检索
- 回复时自动补充相关历史摘要

目标是让机器人不只会“回一句”，而是能记住上下文、延续话题。

### 4. 回复聚焦与 followup

`main.py` 里有专门的回复焦点解析逻辑：

- 对明显的新话题，优先用最新一句作为焦点
- 对“你呢 / 然后呢 / 行吗”这类弱承接句，只补全当前省略对象
- 不回跳更早的旧话题
- `last_target` 是内部锚点概念，不是用户配置项

### 5. 回复发送保护

发送链路做了这些保护：

- 长回复按句子拆分
- 单段发送失败会重试一次
- 多段之间有短延迟
- 单条过长时会按 `reply_settings.max_send_chunk_length` 限制拆分

目的主要是减少平台侧 `sendMsg` 超时和半句输出。

### 6. 主动发言与调度

插件包含两条调度线：

- 常规主动发言和记忆分析
- 新闻自拍任务

常规主动发言会依据最近消息密度、话题相关性和记忆状态判断是否插话。

新闻自拍任务会抓取新闻、判断大新闻、生成文案，并按新闻类型决定是否生成图片。

### 7. 新闻自拍

`news_selfie.py` 是新闻自拍主链路，职责包括：

- 抓取新闻源并解析文章
- 用 LLM 判断哪些新闻值得发送
- 生成第一人称“在现场”的自拍文案
- 获取新闻主图
- 读取当前 skill 目录中的形象资源
- 生成或合成带角色形象的新闻自拍图

当前规则是：

- 消极新闻，如洪水、火灾、灾难等，只发送主图，不额外生成自拍图
- 非消极新闻，优先生成带 bot 形象的自拍式图片
- 自拍图尽量像游戏周边透卡，bot 形象偏左下角或右下角
- 外观来源不是写死角色，而是来自配置指定的 skill 目录

## skill 说明

skill 的入口由 `config.yaml` 的 `skill_settings.active_skill_file` 决定。

当前默认是：

- `skills/薇塔.skill/SKILL.md`

插件会按这个文件定位 skill 目录，并读取：

- `SKILL.md`
- `skill.gif`
- `images/*`

因此：

- 当前看起来像薇塔，是因为配置指向了薇塔 skill
- 机制本身并没有写死薇塔
- 如果以后切换 skill，新闻自拍形象也会跟着切换

## 配置要点

当前比较关键的配置项：

- `plugin_settings.enabled`
- `group_scope.enabled`
- `group_scope.group_whitelist`
- `skill_settings.active_skill_file`
- `skill_settings.bot_names`
- `skill_settings.direct_wake_question_patterns`
- `message_reaction_settings.*`
- `memory_settings.*`
- `probability_settings.*`
- `summary_settings.rollup_time`
- `reply_settings.max_send_chunk_length`
- `proactive_talk_settings.enabled`
- `proactive_talk_settings.cooldown_seconds`
- `proactive_talk_settings.failure_retry_seconds`
- `news_selfie_settings.enabled`

- `image_settings.enabled`
- `image_settings.max_image_description_length`

## 日志风格

插件主要靠日志排障，常见前缀有：

- `[agentic_memory]`
- `[news_selfie]`
- `[scheduled_proactive]`

常见日志内容如下：

### 消息入口

- 收到群消息、群号、发送者、是否 @、是否引用、是否命中名字
- 图片 / 表情 / 引用的提取情况
- 图片识别是否成功、描述是否为空

### 记忆和分析

- 长窗口是否达到阈值
- 是否进入分析流程
- 摘要结果、话题、是否值得插话
- 记忆归纳与动态事件更新情况

### 回复发送

- 回复是否被去重拦掉
- 是否被输出冷却拦掉
- 长回复拆分情况
- 发送失败是否重试成功
- `sendMsg` 超时或底层失败原因

### 新闻自拍

- 新闻抓取结果
- 大新闻评估结果
- 是否判定为消极新闻
- 是否只发送主图
- 是否生成自拍图或合成图
- skill 参考图编码情况
- 图片生成 API 调用与返回情况

## 当前状态总结

这个插件已经具备一条比较完整的群聊智能体链路：

- 能接收并理解群消息
- 能识别图片并把图像语义加入回复
- 能做即时回复、长期记忆和主动插话
- 能定时发新闻自拍
- 能从 skill 中读取角色设定和外观资源

当前重点优化方向是：

- 消息入口更干净，避免事件污染正文
- 图片识别更稳、更完整
- 回复更自然，不回跳旧话题
- 新闻自拍更贴近 skill 形象
- 消极新闻不过度娱乐化

## 后续接手建议

如果后续新会话要继续维护这个插件，建议优先看这几处：

1. `main.py` 的群消息入口和回复焦点逻辑
2. `news_selfie.py` 的新闻筛选、图片生成和 skill 合成逻辑
3. `config.yaml` 的 `image_settings`、`news_selfie_settings`、`reply_settings`
4. 日志前缀 `[agentic_memory]` 和 `[news_selfie]` 的行为是否符合预期

## 备注

- 当前默认 skill 是薇塔，但实现并未写死这个角色
- 新闻自拍和视觉生成会尽量读取配置指定的 skill 目录资源
- 这个文件的用途是给后续会话快速补上下文，不是对外文档

## 近期修复进展

### 已完成

- 修复新闻自拍发送时错误把 `group_id` 当作伪 session 的问题，发送时只使用真实缓存会话。
- 修复回复清洗逻辑，允许 `...`、`..`、`……` 这类省略号式兜底回复直接通过，不再被清成空。
- 给即时回复增加了“半句截断”检测和一次轻量重试，减少明显的断句碎片回复。
- 把主动发言日志整理成中文结构化总日志，增加了主题摘要、LLM 把握度、话题匹配度分、触发概率、随机抽样、最终发言等字段。
- 把大部分运行时英文日志替换成中文，并统一成更直观的固定模板。
- 把新闻自拍任务状态持久化到数据库，避免重启后当天重复执行。
- 把引用机器人消息纳入直接跟随判定，减少“明明引用了 bot 但没反应”的情况。

### 当前仍需继续优化

- 还有少量运行日志或旧前缀可能混在输出里，主要集中在少数新闻自拍分支和个别消息入口分支，需要继续收尾统一。
- 图片下载失败较常见，当前日志已经能看出失败，但还需要进一步判断是网络问题、链接失效还是平台侧限制。
- 目前引用消息已经能直接触发跟随，但对“短句承接”“换话题后的旧话题回拉”仍有边界情况，需要继续收紧 `_resolve_reply_focus()` 和 `_is_weak_followup_message()`。
- 新闻自拍任务虽然已做持久化，但启动时的调度和状态恢复还需要继续观察，确认所有场景都不会重复触发。

### 下个会话建议优先看

1. `main.py` 的 `on_group_message()`、`_is_sender_followup_to_bot()`、`_resolve_reply_focus()`、`_is_weak_followup_message()`。
2. `main.py` 的 `_cron_news_selfie()` 和 `db_manager.py` 的新闻自拍任务状态表。
3. `news_selfie.py` 的抓取、评估、下载、生成、发送日志，继续统一成完全中文模板。
4. 重点观察日志里的 `[agentic_memory]`、`[新闻自拍]`、`[scheduled_proactive]` 前缀是否已经足够清晰。

### 近期日志现象备注

- 图片消息在网络不可用或临时下载失败时，会出现“图片下载失败，已跳过”的日志，这是当前正常保护行为，不一定表示逻辑错误。
- 有些群友会发引用消息或短承接句，例如“？”“草你”“终于”等，这类消息现在更容易被识别为跟随或直接触发，但仍有边界情况。
- 新闻自拍有时会在一次启动周期里先进入管道，再因为状态已持久化而跳过重复执行，这是调度线程先启动、状态检查后置导致的现象，后续可再优化日志顺序和触发门禁。
