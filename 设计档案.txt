插件设计方案（当前实施版）

目标
1. 让插件在群里更像一个真实在线的群成员，而不是只在积攒到固定条数后才突然出现。
2. 在不修改 AstrBot 主项目的前提下，只通过改写 astrbot_plugin_01 完成能力增强。
3. 所有行为阈值、窗口大小、冷却、模型配置、白名单、特殊回复规则等参数全部外置到 config.yaml。
4. 插件仅在配置白名单中的群生效，非白名单群既不看消息，也不回复消息。
5. 插件内部将“后台记忆/总结”和“前台说话”分离，可按场景路由到不同模型。

一、作用范围与边界
1. 仅修改目录：astrbot_plugin_01/
2. 不修改 AstrBot 主项目任何代码。
3. 不在代码中写死业务参数，所有可调参数统一配置化。
4. 插件应支持兼容模式：
   - 模式 A：继续使用 AstrBot 当前默认 provider。
   - 模式 B：插件内自带多模型路由，按不同功能分别调用不同模型。

二、群作用范围控制
1. 插件增加 group_scope 配置。
2. 支持白名单模式：
   - 仅当群号在 group_scope.group_whitelist 中时，插件才处理该群消息。
3. 非白名单群：
   - 不进入消息缓冲区；
   - 不触发即时回复；
   - 不触发记忆总结；
   - 不触发主动插话；
   - 不更新群友档案。

三、说话机制
A. 即时回应链路（被动响应）
1. 下列场景应优先于 buffer_threshold，立即进入回应判定：
   - 有人 @ bot；
   - 文本明确提及 bot 名字、昵称、别称；
   - 命中“提问 bot”的文本模式；
   - 命中特殊意图规则；
   - 其他后续可扩展的直接唤醒条件。
2. 即时回应所需输入上下文由以下几部分组成：
   - 当前触发消息；
   - 最近若干条群聊原始消息；
   - 最近若干条段落总结；
   - 相关群友档案；
   - 当前 skill 内容。
3. 即时回应不应再依赖 buffer_threshold。
4. 即时回应触发后，该条消息仍然必须进入缓冲区，保证记忆连续性。

B. 短窗口感知链路（轻量反应）
1. 为每个群维护一个短反应窗口。
2. 当最近若干条消息持续围绕 bot 展开，但不一定是标准 @ 时，可触发一次轻量回应判定。
3. 短窗口触发独立于长窗口总结机制。
4. 是否启用、窗口大小、最小命中次数、冷却时间均由配置决定。

C. 长窗口观察链路（后台观察与随机插话）
1. 每个群维护一个长消息缓冲区。
2. 当新增消息累计达到 buffer_threshold，并结合 overlap_count 形成滑动窗口时，触发后台记忆分析。
3. 后台分析职责：
   - 生成段落总结；
   - 抽取群友固定档案更新；
   - 抽取群友动态事件；
   - 判断是否为重大事件；
   - 判断是否命中当前 skill 偏好；
   - 生成可供插话的话题切入点。
4. 插话规则：
   - 重大事件：强制触发；
   - 命中偏好：使用偏好加权概率；
   - 普通场景：使用基础概率；
   - 命中触发但无切入话题时，记录日志但不发言。
5. 长窗口分析与即时回应解耦：
   - 即时回应负责“被喊到就回”；
   - 长窗口负责“长期观察、记忆总结、偶尔主动插话”。

D. 主动行为链路（预留扩展）
1. 支持定时主动发言配置。
2. 配置项应支持：
   - 是否启用；
   - 时间窗口；
   - 候选话题列表；
   - 每群每日最多触发次数；
   - 冷却时间。
3. 当前实现若未完整接入，也必须在设计和配置中预留。

四、特殊回复规则
1. 插件支持 special_replies 配置列表。
2. 每条规则至少包含：
   - rule_id；
   - enabled；
   - priority；
   - trigger_mode；
   - patterns；
   - candidate_replies；
   - required_mention；
   - required_name_hit；
   - required_question_mark；
   - cooldown_seconds；
   - scope（如 all_groups / whitelist_only）。
3. 规则命中后可直接绕过 LLM，优先返回预设口语化短回复。
4. 本次必须支持以下两类场景，并且内容配置化，不在代码中写死：
   - 若被提问“怎么还不睡”“怎么还没睡”“为什么还不睡”之类，回复“失眠了”“睡不着”“还没困”等；
   - 若被提问是否买某类周边，直接回复“买！”“当然买”“这还用问，买”等。
5. 特殊回复需支持候选回复随机选取、独立冷却和是否要求命中 bot 唤醒条件。

五、发言去重机制
1. 插件支持 reply_dedup 配置。
2. 默认关闭。
3. 开启后：
   - 记录每个群最近若干条 bot 回复；
   - 计算文本归一化后是否重复；
   - 命中重复时可放弃本次回复或重新请求一次。
4. 去重窗口大小、相似判定策略、是否仅对主动插话生效，均由配置控制。

六、记忆与存储机制
A. 聊天总结
1. 段落总结：
   - 由长窗口分析产生；
   - 存储每次缓冲分析后的总结；
   - 主要服务于近期上下文回顾。
2. 一天总结：
   - 每日定时将前一天段落总结折叠成一天总结；
   - 使用分块归并，避免一次性塞入过长上下文。
3. 老旧历史：
   - 定期把较老的一天总结再次浓缩为长期历史记忆。
4. 所有摘要字数、chunk_size、保留策略、执行时间全部配置化。

B. 群友档案
1. 固定档案：
   - 年龄段、职业、爱好、性别、昵称等可长期保留的信息。
2. 动态事件：
   - 最近发生的小事和大事。
3. 动态事件达到上限后，按配置触发滚动压缩。
4. 滚动压缩的字数限制、事件上限等均由配置控制。

七、LLM 角色隔离
1. 插件内定义多个角色，不同功能可使用不同模型：
   - chat：群聊说话；
   - analysis：后台分析；
   - compression：摘要压缩；
   - event_merge：旧动态事件合并。
2. 配置中支持 llm_settings.mode：
   - astrbot_default：全部走 AstrBot 默认 provider；
   - plugin_router：插件自己按角色访问不同模型。
3. plugin_router 模式下，每个角色单独配置：
   - provider_type；
   - base_url；
   - api_key；
   - model；
   - temperature；
   - top_p；
   - max_tokens；
   - timeout_seconds；
   - extra_headers（可选）。
4. 插件必须有统一的角色路由入口，业务代码不得直接散落多套调用逻辑。

八、配置化原则
以下内容必须全部放进 config.yaml，而不是写死在代码中：
1. 群白名单；
2. bot 名字、昵称、别称；
3. 即时回应最近消息条数；
4. 即时回应带入的最近段落总结条数；
5. 短反应窗口大小、命中次数；
6. 各类冷却时间；
7. buffer_threshold、overlap_count；
8. 概率参数；
9. 摘要压缩 chunk_size、字数提示；
10. 各角色模型配置；
11. 特殊回复 patterns 与 candidate_replies；
12. 去重开关及窗口；
13. 定时主动发言配置；
14. 所有长度、次数、阈值类参数。

九、回复生成原则
1. 回答应口语化、简短、像群友。
2. 提到历史时，只能引用当前记忆系统中真实保留的内容。
3. 超出记忆边界时，应表现为“不记得了”“忘了”“没印象了”等，而不是编造。
4. 涉及他人时，应参考该人的档案与动态事件。
5. 特殊回复优先级高于常规 LLM 回复。
6. 即时回应应快于长窗口分析，优先保证“像人在场”。

十、实现顺序
1. 先重写 config.yaml，使所有参数显式可配。
2. 改写主消息处理流程，引入：
   - 白名单判断；
   - 即时回应；
   - 特殊回复；
   - 短窗口感知；
   - 长窗口缓冲分析。
3. 增加插件内 LLM 路由层。
4. 改写回复生成逻辑，接入最近段落总结条数、档案、技能和去重配置。
5. 保留并兼容现有 SQLite 存储结构。
6. 最后做格式化、检查与回归验证。

十一、验收标准
1. 非白名单群完全不生效。
2. 白名单群中，被 @ 后可立即回复。
3. 白名单群中，被名字提及时可立即回复。
4. 白名单群中，命中特殊回复规则时优先按规则回应。
5. 即时回应不再依赖 buffer_threshold。
6. 即时回应后消息仍进入记忆缓冲。
7. 说话、分析、压缩、事件合并可分别配置不同模型。
8. 发言去重默认关闭，但可通过配置开启。
9. 代码只修改 astrbot_plugin_01，不修改 AstrBot 主项目。

十二、默认建议
1. 白名单默认空列表，避免误在全部群生效。
2. 发言去重默认关闭。
3. 特殊回复默认开启。
4. 即时回应与主动插话使用不同冷却。
5. 总结类模型优先低温、稳定输出；聊天类模型优先更自然口语化输出。
